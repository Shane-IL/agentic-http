Agent-native metadata
for REST APIs

Agentic-HTTP lets your existing API tell AI agents what it does — side effects, retry safety, reversibility, and what to call next. Inline. No new infrastructure. Zero impact on non-agent clients.

Positioning

Not a replacement for OpenAPI.

OpenAPI describes structure — endpoints, parameters, schemas. Agentic-HTTP describes behavior — what happens when an agent calls your API.

OpenAPI tells agents:

  • What endpoints exist
  • What parameters they accept
  • What the response body looks like
  • Which HTTP methods are supported

Agentic-HTTP tells agents:

  • Is it safe to auto-retry on failure?
  • What side effects does calling this trigger?
  • Can the action be reversed?
  • What should I call next in the workflow?
  • How do I recover from this specific error?

The two are complementary. OpenAPI is the structural contract, written once at design time. Agentic-HTTP is the behavioral contract, delivered inline at call time — from the source that knows best. The openapi-to-agentic tool generates Agentic-HTTP stubs directly from an existing OpenAPI spec.

Protocol

One header. One extra field.

No new transport. No wrapper service. No changes to existing clients.

1

Agent signals intent

The agent adds X-Agent-Client: true and X-Agent-Protocol: 1.0 to its request. Regular clients never send these — they're completely unaffected.

2

Server checks scope

The endpoint verifies the caller has the agentic:read OAuth scope. Unauthenticated callers receive a standard response with no metadata.

3

Response includes agent-meta

The endpoint appends an agent-meta object to its existing JSON response. The data payload is unchanged. No new routes, no new transport.

Example

POST /users — with agent metadata

The response is identical for non-agent callers. Agent callers receive the full behavioral context inline.

Request
Response
POST /users HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...
X-Agent-Client: true
X-Agent-Protocol: 1.0
Content-Type: application/json

{ "email": "alice@example.com", "name": "Alice" }
{
  "data": {
    "id": "usr_1a2b3c",
    "email": "alice@example.com",
    "status": "pending-verification"
  },
  "agent-meta": {
    "version": "1.0",
    "intent": "Creates a new user account. Triggers a verification email to the address provided.",
    "effect": "write",
    "reversible": false,
    "idempotent": false,
    "retry-safe": false,
    "side-effects": ["sends-email", "creates-db-record"],
    "preconditions": ["email must not already exist in the system"],
    "typical-next": ["/users/usr_1a2b3c/verify"],
    "error-guidance": {
      "409": "User already exists. Do not retry. Use GET /users?email= to retrieve the existing record.",
      "422": "Validation failed. Inspect errors[] for field-level details before retrying."
    }
  }
}
Spec

agent-meta field reference

Required fields ensure every agent-meta envelope carries the minimum context for safe autonomous operation.

Field Type Description
version string Required Schema version. Currently "1.0".
intent string Required Plain-language description of what this endpoint does, written for an agent reasoning about whether to call it.
effect enum Required One of read, write, delete, mixed.
reversible boolean Required Whether the effects of this call can be undone via another API call.
idempotent boolean Required Whether calling this endpoint multiple times with the same input produces the same result.
retry-safe boolean Required Whether an agent may automatically retry on transient failure.
side-effects string[] Optional Machine-readable labels for observable side effects. Lowercase hyphenated, e.g. sends-email.
preconditions string[] Optional Natural-language conditions that should be true before calling this endpoint.
typical-next string[] Optional Ordered list of endpoint paths typically called after this one. Enables workflow chaining without external docs.
error-guidance object Optional Map of HTTP status code strings to agent-readable recovery instructions for this specific endpoint.
extensions object Optional Namespaced extension fields. Keys must be prefixed x-{namespace}-.
Reference Implementations

Drop it into your stack.

Middleware and decorators for three frameworks. Same behavior, idiomatic to each runtime.

🟨

Express / Node.js

Per-route middleware factory. Wraps res.json to append agent-meta inline.

 🐍

FastAPI / Python

Decorator + async middleware. Integrates with FastAPI dependency injection and response models.

 🔴

NestJS / TypeScript

@AgenticMeta() decorator + interceptor. Module-level setup via AgenticHttpModule.forRoot().
